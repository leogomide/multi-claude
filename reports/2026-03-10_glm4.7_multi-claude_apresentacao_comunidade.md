## Executivo

Este relatório consolidou análises técnicas, de UX e de mercado para preparar uma apresentação do projeto **multi-claude** em uma live de comunidade de desenvolvimento com IA. O projeto é uma CLI tool que gerencia múltiplos provedores de API e inicia o Claude Code com as variáveis de ambiente corretas.

---

## 1. Resumo do Projeto

### O que é o multi-claude?

CLI tool para gerenciar múltiplos provedores de API e iniciar o Claude Code com as environment variables corretas. O usuário roda `mclaude` para selecionar um provedor (o gerenciamento de providers é feito dentro da TUI).

### Stack Tecnológico

| Componente | Tecnologia | Justificativa |
|------------|------------|----------------|
| Runtime | Bun | Performance, startup time, TS direto sem build |
| UI | Ink + React | TUI interativa no terminal |
| Validação | Zod | Type safety |
| Internacionalização | Rosetta | EN, PT-BR, ES |

### Arquitetura Principal

```
cli.ts (Entry Point)
  ├─> tui-process.ts (Isolamento Ink/React)
  │   └─> app.tsx (Render Ink)
  │       └─> UnifiedApp.tsx (Router Principal)
  │           ├─> AppShell.tsx (Layout)
  │           └─> Páginas (MainMenu, StartClaudeFlow, etc.)
  └─> runner.ts (Spawn do Claude)

Configuração:
  ├─> config.ts (Persistência em ~/.multi-claude/)
  ├─> providers.ts (Templates de providers)
  └─> services/ (APIs externas)
```

## 2. Problemas Resolvidos (Dores do Desenvolvedor)

### Dor Principal: Fragmentação de provedores Claude Code

O Claude Code nativo suporta apenas Anthropic (OAuth) ou configuração manual via environment variables. Desenvolvedores que querem usar provedores alternativos enfrentam:

1. **Gerenciamento manual de variáveis** - `ANTHROPIC_API_KEY`, `ANTHROPIC_BASE_URL`
2. **Alternância manual entre chaves API** - Sem memória de configuração
3. **Apenas uma conta OAuth** - Claude Code nativo suporta apenas uma
4. **Configuração repetitiva** - Toda sessão requer setup
5. **Falta de visibilidade** - Sem métricas de custo/tokens em tempo real

### Comparativo: Antes vs Depois

| Problema | Sem mclaude | Com mclaude |
|----------|-------------|-------------|
| Multi-conta Anthropic | Apenas 1 conta OAuth | Múltiplas contas isoladas |
| Troca de provider | `export ANTHROPIC_API_KEY=...` | `mclaude` → seleciona → done |
| Gerenciamento de chaves | Expostas em shell/.env | Seguras em config.json |
| Instalações isoladas | Apenas `~/.claude/` | Múltiplas instalações |
| Visibilidade de custos | Sem métricas | Status line real-time |

## 3. Diferenciais Únicos

1. **Multi-provider unificado** - 11+ provedores em uma interface
2. **TUI com Ink/React** - Interface interativa no terminal
3. **Instalações isoladas** - Cada provider tem sua config, MCP servers, histórico
4. **Status line customizável** - 7 templates com métricas em tempo real
5. **Headless mode** - Para scripting/automação com flags CLI
6. **OAuth flow** - Suporte a múltiplas contas Anthropic simultâneas
7. **Model discovery dinâmico** - OpenRouter, Requesty, Ollama buscam via API
8. **Bundled skill** - Ensin outros agentes a usarem mclaude

## 9. Perguntas Esperadas + Respostas

| Pergunta | Resposta |
|----------|----------|
| Como funciona a integração com Claude Code? | Environment variables (ANTHROPIC_BASE_URL, ANTHROPIC_AUTH_TOKEN). |
| Como descobrem os modelos disponíveis? | Alguns providers têm API de modelos (OpenRouter). Outros têm default no template. |
| Como funcionam as instalações isoladas? | Cada instalação é ~/.multi-claude/installations/<id>/. Usamos CLAUDE_CONFIG_DIR. |
| Por que Bun e não Node? | Performance, startup time, DX. Bun roda TS direto sem build step. |
| Por que open source? | Comunidade de IA precisa de ferramentas flexíveis. Queremos contribuições. |
| Qual o roadmap? | Mais providers, melhorias na status line, mais integrações MCP. |

## 11. Insights Técnicos

### Padrões de Design Identificados

1. **Facade Pattern** - `api-models.ts` unifica APIs diferentes
2. **Strategy Pattern** - `configureEnv()` customizado por provider
3. **Template Method** - Fluxos de configuração com steps
4. **Repository Pattern** - `config.ts` abstrai persistência

### Decisões Técnicas Interessantes

1. **Isolamento de Processo** - Ink/React não podem coexistir com spawn do Claude na mesma thread
2. **Main Loop Pattern** - Usuário retorna ao menu automaticamente após sessão
3. **Status Line Injection** - Script Python dinâmico injetado via `--settings`
4. **OAuth Credentials** - Armazenadas em `~/.multi-claude/accounts/<provider-id>/`

## Conclusão

O multi-claude resolve um problema real para usuários avançados do Claude Code que precisam de:
- Flexibilidade na escolha de provedores
- Isolamento entre contextos de trabalho
- Gerenciamento centralizado de configurações
- Visibilidade de uso e custos

O projeto posiciona-se como uma camada de orquestração sobre o Claude Code, não um substituto, mantendo compatibilidade total com todas as funcionalidades nativas.
